# システムプランナー研修資料
**対象：初級自動化エンジニア（入社1ヶ月～）**  
**目標：上級自動化エンジニア認定に必要なプランナースキルの習得**

---

## 1. システムプランナーってなに？

### 1.1 基本的な役割

一言でいうと、**「自動化という武器を、いつ、どこで、どう使うか決める軍師」**です。

単にコードを書いたりMagicPodでステップを作ったりするのが「職人（エンジニア）」なら、プランナーは「どのテストを自動化すれば一番効果が高いか？」を戦略的に考えます。

**対比で理解しよう：**

| 役割 | エンジニア（職人） | プランナー（軍師） |
|------|-------------------|-------------------|
| 関心事 | 「このスクリプト、動くように書けたぞ！」 | 「そのテスト、毎日実行する価値はある？メンテナンスの手間の方が大きくない？」 |
| 判断基準 | 技術的に実装可能か | ビジネス的に価値があるか |
| アウトプット | 動作するテストコード | 自動化計画・ROI分析 |

### 1.2 なぜプランナーが必要なのか？

**よくある失敗例：**
```
❌ 悪い例：「この画面、全部自動化しました！」
   → 結果：メンテナンスに毎週2時間、実行されるのは月1回だけ...
   
✅ 良い例：「この画面は月1回しか使われないので、手動テストのままにします。
          代わりに、毎日実行するログイン処理を優先的に自動化しました」
   → 結果：毎日30分の工数削減に成功！
```

**新人さんが陥りがちな罠：**
- 「自動化できるから」という理由だけで何でも自動化してしまう
- メンテナンスコストを考えずに複雑なテストを作ってしまう
- 実行頻度や重要度を考慮せずに優先順位をつけてしまう

**プランナーの思考回路：**
1. 「このテストは何回実行される？」（頻度）
2. 「失敗したときの影響は？」（重要度）
3. 「自動化にどれくらい時間がかかる？」（工数）
4. 「メンテナンスは大変？」（保守性）
5. 「手動でやるより本当に楽になる？」（ROI）

## 2. どんな知識が必要なの？

プランナーには、技術と運用の両方の視点が求められます。

### 2.1 テスト設計の知識

**何をテストすべきか（テストスイートの選定）**

- **境界値分析：** 入力値の境界（0、-1、最大値など）をテストする
- **同値分割：** 同じ結果になるグループに分けて代表値をテストする
- **デシジョンテーブル：** 条件の組み合わせを表で整理する

**新人向けポイント：**
「全部のパターンをテストする」のは理想ですが、現実的には不可能です。プランナーは「どのパターンを優先すべきか」を判断します。

**優先順位の付け方（例）：**
1. **クリティカルパス：** ユーザーが必ず通る操作（ログイン、決済など）
2. **高頻度操作：** 毎日何度も使われる機能
3. **過去に不具合が多い箇所：** バグの温床になっている画面
4. **複雑なロジック：** 手動でテストすると時間がかかる処理

### 2.2 ツールの特性（MagicPodなど）

**「できること」と「苦手なこと」の把握**

| ツールの得意分野 | ツールの苦手分野 |
|----------------|-----------------|
| 繰り返し実行されるテスト | 1回しか実行されないテスト |
| 安定した画面構造 | 頻繁に変わる画面 |
| シンプルな操作フロー | 複雑な条件分岐 |
| データドリブンテスト | アドホックな探索的テスト |

**具体例で理解しよう：**
```
【自動化に向いているケース】
✅ ログイン処理（毎回同じ手順、毎日実行）
✅ 商品検索機能（パターンが明確、高頻度）
✅ 帳票出力（データを変えて複数回実行）

【自動化に向いていないケース】
❌ 新機能のUI確認（仕様が固まっていない）
❌ エラーメッセージの文言チェック（目視の方が早い）
❌ デザインの見た目確認（人の判断が必要）
```

### 2.3 ROI（投資対効果）の考え方

**自動化にかかる工数と、削減できる手動工数の比較**

**ROIの基本公式：**

$$ROI = \frac{削減できる工数（利益）}{自動化にかかる工数（投資）} \times 100$$

**実践的な判断基準：**
- **ROI > 100%：** 自動化すべき（投資以上のリターンがある）
- **ROI = 50〜100%：** 要検討（他の要因も考慮）
- **ROI < 50%：** 自動化しない方が良い

**具体例で計算してみよう：**
```
【ケース1：ログイン処理の自動化】
- 手動実行時間：5分/回
- 実行頻度：1日3回 × 20営業日 = 60回/月
- 月間削減時間：5分 × 60回 = 300分 = 5時間

- 自動化工数：8時間（初回作成）
- メンテナンス：1時間/月

→ 2ヶ月目で元が取れる！（8時間 < 5時間 × 2ヶ月）
→ ROI = 良好 ✅

【ケース2：年1回の帳票出力】
- 手動実行時間：30分/回
- 実行頻度：年1回
- 年間削減時間：30分

- 自動化工数：10時間
- メンテナンス：2時間/年（仕様変更対応）

→ 20年かかっても元が取れない...
→ ROI = 悪い ❌
```

### 2.4 プロジェクト管理

**進捗を数字で捉え、周囲に説明する能力**

**プランナーが管理すべき指標（KPI）：**

1. **自動化カバレッジ：** 全テストケースのうち自動化済みの割合
```
   カバレッジ = (自動化済みTC数 ÷ 全TC数) × 100%
   目標値：60〜80%（100%を目指すのは非現実的）
```

2. **実装進捗率：** 計画に対する実績
```
   進捗率 = (完了TC数 ÷ 予定TC数) × 100%
```

3. **テスト実行時間：** 自動化によりどれだけ短縮できたか
```
   短縮時間 = 手動実行時間 - 自動実行時間
```

4. **メンテナンス率：** どれだけ修正が発生しているか
```
   メンテナンス率 = (修正が必要だったTC数 ÷ 全TC数) × 100%
   目標値：20%以下（これを超えると保守コストが高すぎる）
```

**報告書の例：**
```
【今月の実績】
- 自動化完了：25件 / 目標30件（進捗率：83%）
- 自動化カバレッジ：65%（前月比+5%）
- 削減工数：今月20時間の手動工数を削減
- メンテナンス：3件の修正が発生（メンテナンス率：12%）

【来月の計画】
- 残り5件の実装完了
- 新規プロジェクトAの自動化計画策定
```

## 3. 実績ベースの見積もり術（MagicPod編）

見積もりは「勘」ではなく、**「過去の数字（実績)」**で語るのが一番説得力があります。あなたが実践している「1ステップあたりの速度」から算出する方法は、非常に理にかなったプロの手法です。

研修資料用に、数式を整理してみましょう。

### 3.1 STEP 1：実装スピード（単位能力）を出す

まずは、自分たちが**1ステップ作るのに何分かかっているか**を知ることから始めます。

$$V_{step} = \frac{T_{avg}}{S_{avg}}$$

**記号の意味：**
- $V_{step}$：1ステップあたりの実装速度（分/ステップ）
- $T_{avg}$：1テストケース（TC）の作成にかかった平均時間（分）
- $S_{avg}$：1TCあたりの平均ステップ数（ステップ）

**具体例で計算してみよう：**
```
【実績データ（過去10件のテストケースから算出）】
- TC1：30分、15ステップ
- TC2：45分、20ステップ
- TC3：20分、10ステップ
...
- TC10：40分、18ステップ

【計算】
T_avg = (30 + 45 + 20 + ... + 40) ÷ 10 = 35分
S_avg = (15 + 20 + 10 + ... + 18) ÷ 10 = 16ステップ

V_step = 35分 ÷ 16ステップ = 2.19分/ステップ

→ つまり、このチームは「1ステップ作るのに約2.2分かかる」ということ
```

**新人向けポイント：**
- 最初は実績データがないので、まずは5〜10件のテストケースを作ってみる
- 慣れてくると速度が上がるので、定期的に（月1回程度）見直す
- 複雑なステップは時間がかかるので、平均値だけでなくバラつきも確認する

### 3.2 STEP 2：1日あたりの「生産限界」を出す

次に、**1人日（実働6時間）で何件のテストケースが作れるか**を計算します。

$$C_{day} = \frac{H_{day}}{V_{step} \times S_{avg}}$$

**記号の意味：**
- $C_{day}$：1日の実装可能件数（TC数/日）
- $H_{day}$：1日の稼働時間（例：6.0時間 = 360分）

**具体例で計算してみよう：**
```
【前提】
- V_step = 2.19分/ステップ（STEP 1で計算済み）
- S_avg = 16ステップ/TC（STEP 1で計算済み）
- H_day = 6時間 = 360分

【計算】
1TC作成時間 = 2.19分/ステップ × 16ステップ = 35分/TC

C_day = 360分 ÷ 35分/TC = 10.3件/日

→ つまり、このチームは「1日で約10件のテストケースを作れる」ということ
```

**新人向けポイント：**
- 「1人日 = 6.0時間」の理由：
  - 8時間 - 会議1時間 - 休憩1時間 = 実働6時間
  - 予期せぬトラブル対応の余裕も確保
- 新人の場合は、さらに×0.7〜0.8倍（習熟度係数）を掛けると安全

**習熟度による補正：**
```
- 新人（1〜3ヶ月）：×0.7倍  → 10.3件 × 0.7 = 7.2件/日
- 中堅（3〜12ヶ月）：×0.9倍 → 10.3件 × 0.9 = 9.3件/日
- ベテラン（12ヶ月〜）：×1.0倍 → 10.3件/日
```

### 3.3 STEP 3：総見積もりを出す

最後に、**対象となるテスト全件を終わらせるのに何日かかるか**を出します。

$$E = \frac{N_{total}}{C_{day}}$$

**記号の意味：**
- $E$：実装見積もり（人日）
- $N_{total}$：テスト対象の総テストケース数

**具体例で計算してみよう：**
```
【プロジェクト情報】
- 総テストケース数：150件
- 担当者：新人エンジニア1名（習熟度0.7）

【計算】
C_day（新人） = 10.3件/日 × 0.7 = 7.2件/日

E = 150件 ÷ 7.2件/日 = 20.8人日

→ 約21営業日（約4週間）かかる見込み
```

**複数人で分担する場合：**
```
【ケース：3人チームで分担】
- ベテラン1名：10.3件/日
- 中堅1名：9.3件/日
- 新人1名：7.2件/日
- チーム合計：26.8件/日

E = 150件 ÷ 26.8件/日 = 5.6人日

→ 約6営業日（約1.2週間）で完了
→ ただし、実際にはレビュー時間や調整が必要なので+20%の余裕を見る
→ 最終見積もり：6日 × 1.2 = 7.2日（約1.5週間）
```

### 3.4 見積もりの精度を上げるコツ

**チェックリスト：**

- [ ] 過去の実績データを使っているか？（勘で見積もっていないか？）
- [ ] 習熟度係数を考慮しているか？
- [ ] 複雑度の違いを考慮しているか？（簡単なTCと複雑なTCを区別）
- [ ] バッファ（余裕）を持たせているか？（+20%推奨）
- [ ] 定期的に見積もりと実績を比較して改善しているか？

**見積もりと実績の記録例：**

| 案件名 | 見積もり | 実績 | 差異 | 原因分析 |
|--------|---------|------|------|---------|
| プロジェクトA | 20人日 | 24人日 | +4人日 | 仕様変更が3回あった |
| プロジェクトB | 15人日 | 13人日 | -2人日 | 想定より単純だった |
| プロジェクトC | 30人日 | 35人日 | +5人日 | 新しいツールで不慣れ |

→ この記録を元に、次回の見積もり精度を向上させる

## 4. プランナーとして忘れてはいけない「＋α」

実績ベースの見積もりは強力ですが、新人エンジニアが陥りがちな落とし穴もあります。資料には以下の「バッファ（余裕）」の考え方も入れておくと、さらにクオリティが上がります。

### 4.1 メンテナンス工数

**画面の仕様が変わったときの手直し時間**

自動化は「作って終わり」ではありません。システムは常に進化するため、テストスクリプトも更新が必要です。

**メンテナンスが発生する主なパターン：**

1. **UI変更：** ボタンの位置や名前が変わった
```
   例：「送信」ボタン → 「確定」ボタンに変更
   → 全てのテストケースで要素の指定を修正
```

2. **仕様変更：** 業務フローが変わった
```
   例：承認フローが1段階から2段階に変更
   → テストステップ自体を追加・削除
```

3. **データ変更：** テストデータの形式が変わった
```
   例：郵便番号の形式が「123-4567」→「1234567」に変更
   → 全てのテストデータを修正
```

**メンテナンス工数の目安：**
```
メンテナンス工数 = 初回実装工数 × 0.2（20%）

【具体例】
- 初回実装：150件のTC、30人日
- 年間メンテナンス：30人日 × 0.2 = 6人日
- 月間換算：6人日 ÷ 12ヶ月 = 0.5人日/月
```

**メンテナンスコストを下げる工夫：**

- **共通部品化：** 繰り返し使う操作はサブルーチン化
```
  悪い例：ログイン処理を100個のTCにコピペ
       → 修正時に100箇所直す必要がある
  
  良い例：ログイン処理を共通部品にする
       → 1箇所直せば全TCに反映される
```

- **保守性の高い設計：** 要素の指定方法を工夫
```
  悪い例：XPath = "/html/body/div[3]/button[2]"
       → 画面構造が変わると即座に壊れる
  
  良い例：ID指定 = "#submit-button"
       → IDが変わらない限り動作する
```

### 4.2 環境構築

**MagicPodの設定や、テストデータ準備にかかる時間**

自動化を始める前に、実行環境やテストデータを整える必要があります。

**環境構築で必要な作業：**

1. **ツール設定：**
   - MagicPodプロジェクトの作成
   - 実行端末（iOS/Android）の準備
   - CI/CD連携の設定
   - 推定時間：2〜5人日

2. **テストデータ準備：**
   - テスト用アカウントの作成
   - マスタデータの投入
   - テスト環境のセットアップ
   - 推定時間：3〜10人日（システム規模による）

3. **ドキュメント整備：**
   - 命名規則の策定
   - フォルダ構成の設計
   - テンプレートの作成
   - 推定時間：1〜2人日

**新人向けポイント：**
```
環境構築は見落とされがちですが、プロジェクト全体の10〜20%の工数がかかります。

【見積もり例】
- 実装工数：30人日
- 環境構築：30人日 × 0.15 = 4.5人日
- レビュー工数：30人日 × 0.1 = 3人日
- メンテナンス（年間）：30人日 × 0.2 = 6人日
─────────────────────────
  合計：30 + 4.5 + 3 + 6 = 43.5人日

→ 実装だけでなく、全体像を見積もることが大切！
```

### 4.3 レビュー工数

**作成したスクリプトが正しいか、チームで確認する時間**

自動化テストも人間が作るので、バグやロジックミスがあります。レビューは品質を保つために必須です。

**レビューで確認すべきポイント：**

1. **機能的な正しさ：**
   - [ ] テストの目的を満たしているか？
   - [ ] テストケース設計は適切か？
   - [ ] 境界値テストは含まれているか？

2. **技術的な正しさ：**
   - [ ] 要素の指定方法は適切か？
   - [ ] 待機時間（wait）は適切に設定されているか？
   - [ ] エラーハンドリングは実装されているか？

3. **保守性：**
   - [ ] 命名規則に従っているか？
   - [ ] コメントは適切に書かれているか？
   - [ ] 共通部品を活用しているか？

**レビュー工数の目安：**
```
レビュー工数 = 実装工数 × 0.1（10%）

【具体例】
- 実装：5時間
- レビュー：5時間 × 0.1 = 0.5時間（30分）

→ 大体、作成時間の10分の1がレビュー時間
```

**効率的なレビューのコツ：**

- **ペアプログラミング：** 作成しながらレビュー（新人におすすめ）
- **チェックリスト：** 確認項目を標準化して見落としを防ぐ
- **自動チェック：** リント（構文チェック）ツールで機械的にチェック

### 4.4 リスクバッファ

**予期せぬトラブルに備える余裕時間**

どんなに綿密に計画しても、予想外のことは起こります。プランナーは常にリスクを想定し、バッファを持たせます。

**よくあるリスク事例：**

| リスク | 発生頻度 | 影響度 | 対策 |
|--------|---------|--------|------|
| 仕様変更 | 高い | 大きい | バッファ+20% |
| ツールの不具合 | 中程度 | 中程度 | 代替手段を用意 |
| 担当者の不在 | 低い | 大きい | 作業の引き継ぎ書を作成 |
| テスト環境の不安定 | 高い | 中程度 | リトライ機能を実装 |

**バッファの持たせ方：**
```
【保守的な見積もり（おすすめ）】
基本見積もり × 1.2（+20%）

【積極的な見積もり（経験者向け）】
基本見積もり × 1.1（+10%）

【楽観的な見積もり（非推奨）】
基本見積もり × 1.0（バッファなし）
→ 新人は絶対にこれをやってはいけない！
```

## 5. 実践演習：見積もりをやってみよう

### 5.1 演習問題

あなたは新しいプロジェクトの自動化計画を立てることになりました。以下の情報を元に見積もりを作成してください。

**プロジェクト情報：**
- 対象システム：ECサイトの注文機能
- 総テストケース数：80件
- あなたの実績データ：
  - 平均作成時間：40分/TC
  - 平均ステップ数：20ステップ/TC
- あなたの経験：入社3ヶ月（習熟度0.8）
- 実働時間：1日6時間

**課題：**
1. 1ステップあたりの実装速度を計算してください
2. 1日の実装可能件数を計算してください
3. 総見積もり（人日）を計算してください
4. バッファ（+20%）を考慮した最終見積もりを出してください

### 5.2 解答例
```
【STEP 1：実装速度】
V_step = T_avg ÷ S_avg
V_step = 40分 ÷ 20ステップ = 2.0分/ステップ

【STEP 2：1日の実装可能件数】
1TC作成時間 = 2.0分 × 20ステップ = 40分/TC
C_day = 360分 ÷ 40分/TC = 9件/日

習熟度補正：9件/日 × 0.8 = 7.2件/日

【STEP 3：基本見積もり】
E = 80件 ÷ 7.2件/日 = 11.1人日 ≒ 12人日

【STEP 4：最終見積もり】
最終見積もり = 12人日 × 1.2（バッファ） = 14.4人日 ≒ 15人日

→ 約3週間（15営業日）で完了予定
```

### 5.3 チェックリスト：見積もり前の確認事項

プランナーとして見積もりを出す前に、以下を確認しましょう。

- [ ] レビュー工数を含めたか？

- [ ] バッファ（余裕）を持たせたか？

## 6. よくある質問（FAQ）

### Q1：見積もりが大幅にズレたらどうすればいい？

**A1：** まず原因を分析し、次回に活かすことが大切です。
```
【分析の視点】
1. 前提条件が変わっていないか？
   - 仕様変更があった
   - ツールのバージョンアップがあった
   - テスト環境が不安定だった

2. 見積もりの根拠が甘かったか？
   - 実績データが少なかった
   - 複雑度を見誤った
   - バッファが足りなかった

3. 実装方法に問題があったか？
   - 無駄な作業をしていた
   - 効率的な方法を知らなかった
   - レビューで大幅な作り直しが発生した
```

**対応策：**
- すぐにステークホルダーに報告（遅れが確定した時点で）
- 原因と対策を説明
- 見積もりデータを更新

### Q2：すべてのテストを自動化すべきですか？

**A2：** いいえ、100%自動化は目指すべきではありません。

**自動化すべきもの（優先度高）：**
- 繰り返し実行されるテスト（回帰テスト）
- 手動では時間がかかるテスト（データドリブンテスト）
- クリティカルな機能のテスト（ログイン、決済）

**手動のままで良いもの：**
- 1回しか実行されないテスト
- 探索的テスト（バグを探す作業）
- ユーザビリティテスト（使いやすさの確認）
- 見た目の確認（デザインレビュー）

**目標カバレッジ：60〜80%**
（残り20〜40%は手動テストで補完）

### Q3：メンテナンスコストを下げるにはどうすればいい？

**A3：** 設計段階で保守性を意識することが重要です。

**具体的な方法：**

1. **共通部品化：**
```
   ログイン処理、検索処理など、繰り返し使う操作は
   サブルーチンとして切り出す
```

2. **データ駆動型テスト：**
```
   テストロジックとテストデータを分離する
   → データだけ変えれば複数パターンをテストできる
```

3. **堅牢な要素指定：**
```
   - ID指定を優先（変わりにくい）
   - XPathは最終手段（変わりやすい）
```

4. **定期的なリファクタリング：**
```
   月1回程度、古いテストコードを見直して改善
```

**A4：** はい、段階的にスキルを身につけていけば可能です。
